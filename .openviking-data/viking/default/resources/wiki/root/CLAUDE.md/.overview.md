# CLAUDE

This directory contains the `CLAUDE.md` project guide for working safely inside a GitNexus-indexed Lab codebase. It focuses on repository navigation, impact-aware code changes, symbol-level refactoring, debugging, and validation workflows. The document is aimed at engineers and agents using GitNexus MCP tools who need to understand code symbols, execution flows, blast radius, and safe commit practices. It emphasizes mandatory steps such as running `gitnexus_impact` before editing symbols, reporting high or critical risk before proceeding, using `gitnexus_detect_changes` before commit, and relying on GitNexus query/context tools instead of generic text search when exploring unfamiliar code. Keywords: GitNexus, code intelligence, impact analysis, blast radius, repository workflow, refactoring, debugging, CLI.

## Detailed Description

### CLAUDE.md `CLAUDE.md`

`CLAUDE.md` is the central developer guide and operating reference for this repository. It defines mandatory workflows for safe, impact-aware modifications in a GitNexus-managed environment. The document is organized into practical sections such as “Always Do,” “Never Do,” “Resources,” and “CLI,” making it easy to follow as a day-to-day protocol.

A major theme of the file is change safety. It instructs users to begin any symbol edit with `gitnexus_impact`, to escalate high- or critical-risk findings before continuing, and to validate changes with `gitnexus_detect_changes` before committing. This makes the guide especially useful for tasks involving refactoring, debugging, or feature updates where unintended side effects matter.

The file also explains how to navigate the codebase effectively. Instead of using broad text searches like `grep` on unfamiliar code, it directs users to GitNexus query and context tools for symbol-aware discovery, execution-flow analysis, and architecture exploration. This reflects a repository workflow built around code intelligence rather than raw text scanning.

Finally, `CLAUDE.md` points to related GitNexus skill files for architecture review, impact analysis, debugging, refactoring, and CLI usage. In practice, this makes the file both a policy document and a launch point for deeper repository understanding. It is best suited for developers, maintainers, and AI agents operating in the codebase who need a concise but strict reference for safe repository actions.
